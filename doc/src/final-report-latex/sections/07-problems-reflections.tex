\chapter{Problem \& Reflections}

Many of the hardest problems in this project were not purely visual and not purely logical. In practice, presentation, interaction, testing, and planning were tightly coupled. Once the interface became more elaborate, every UI change exposed another layer of engineering work: raycast boundaries, runtime layout assumptions, battle-state messaging, or proof that a fix had actually held. This chapter reflects on those issues at technical, project-management, and team-management levels.

\section{Technical Problem}

One recurring technical problem was that visual polish changed interaction behaviour. As the title overlay, themed HUD, and hand dock became more elaborate, previously hidden assumptions in the prototype started to fail. Concrete examples included title overlay content colliding vertically, the top-left prompt stack overlapping turn and energy labels, the hand dock obscuring lower units, and action availability being visible only in the battle log.

The important lesson was that these defects could not be handled with visual tweaking alone. Several bugs existed at the boundary between layout and logic. A panel could look harmless but still intercept clicks; a debug shortcut could speed up testing but bypass the same flow path used by the real HUD; a screenshot could show the board correctly while missing overlay elements entirely.

Another technical limitation came from tooling. Overlay screenshots were not always trustworthy for `Screen Space Overlay` verification in this setup. The practical response was to split validation responsibility: hierarchy assertions and runtime tests for structural proof, and manual smoke for final overlay readability.

\section{Project Management}

The main project-management difficulty was scope control. The original idea could easily have expanded into a larger tactics game with more cards, more enemy types, more maps, and some form of progression system. In practice, every time the team widened the presentation or interaction layer, it also created more testing and debugging work. That made it necessary to keep returning to the same rule: finish one battle slice properly before adding more content.

This affected planning throughout the project. Some work that initially looked like polish, such as title flow, result flow, disabled-card feedback, or health-bar visibility, later turned out to be necessary for a complete and readable submission. As a result, backlog decisions were not based only on what sounded interesting to add next. They were based on which issue most directly blocked the game from being understandable, stable, or demonstrably complete.

Testing and evidence collection also became planning tools. When a feature had no clear validation path, that was treated as a warning sign that the implementation was still too vague. This is one reason the project stopped short of adding wider content late in development. It was more valuable to close visible usability defects with proof than to keep extending the ruleset without the same level of control.

\section{Team Management}

Our team coordination was not great at the beginning of the project. For the first few weeks, most of the communication happened through WeChat group messages, and files were shared as loose Word documents with a naming convention we agreed on in Week 1 (see Appendix B.3). At that point we did have a GitHub repository set up, but in practice nobody was really committing regularly. Hongshuo and Cheng were doing most of the coding work locally, and Guangjun was writing requirements documents in separate `.docx` files. Zehan was working on testing but did not have much to test until the prototype was more stable. So the actual project state lived in people's local machines and chat history rather than in one traceable place.

This caused confusion around the middle of the project. After the second weekly submission, we made a more serious effort to move everything onto GitHub with proper branches and pull requests, and also started collecting screenshots, test records, and meeting notes into the documentation tree in the repository. Around the same time, we moved our report writing from Word documents to Overleaf so that everyone could edit the same LaTeX source simultaneously. Before that switch, the report drafting process was messy. One person would write a section in Word, send it to the group chat, and then another person would copy-paste it into whichever version they had open. Merging those scattered drafts often took longer than the writing itself. With Overleaf we could at least see each other's edits in real time, which reduced duplicated paragraphs and version mismatches. It also forced us to learn LaTeX along the way, which slowed things down at first but made the final formatting much more consistent than the earlier Word submissions.

That said, the switch to a proper workflow for both code and documentation came later than it should have. Because of this, some of our early meeting records and planning decisions are only preserved as recovered notes rather than as proper version-controlled documents, as described in Appendix B.1. If we were to start over, we would enforce a repository-first workflow from Week 1 and set up Overleaf at the same time instead of halfway through. The other thing we would change is having clearer code ownership earlier. For most of the project, it was not always obvious who was responsible for which scripts, and this sometimes led to duplicated effort or parts of the code that nobody felt confident enough to modify.
