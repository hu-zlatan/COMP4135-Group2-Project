\chapter{Test \& System Evaluation}

The validation approach does not rely on a single type of evidence. Instead, it combines direct EditMode tests for stable logic, PlayMode smoke for whole-scene readiness, and manual smoke for visual composition and overlay-heavy UI behaviour.

\section{Test Strategy}

The core strategy is \textit{TDD-first where behaviour is deterministic, smoke-first where the result is primarily visual}. This follows the practical value of test-driven development: make behaviour explicit, protect it with fast regression checks, and then refactor with greater confidence \cite{beck2003tdd}. Turn flow, deck lifecycle, movement constraints, damage handling, and battle-end state all have explicit success conditions and can be tested directly. By contrast, title-panel readability, overlay composition, and the visual relationship between the board and the hand dock cannot be reduced safely to screenshot-perfect automation in the current project setup.

\section{Issue Discovery and Closure Loop}

The most convincing part of the final validation package is not a single score or count. It is the way concrete problems were discovered, translated into engineering work, and then closed with evidence. The project repeatedly followed the same loop:

\begin{enumerate}
    \item a visible problem was found through playtesting or smoke;
    \item the deterministic part of that problem was isolated;
    \item a regression test or structural assertion was added where possible;
    \item runtime code was fixed;
    \item the fix was checked again through rerun or smoke.
\end{enumerate}

\section{Evaluation and Technical Self-Critique}

The delivered build satisfies the main goal of the project: it provides a complete and readable tactical battle loop rather than only a technical prototype. The player can begin from a title screen, enter battle, select units and cards, understand the turn state through the HUD, observe enemy response, and reach a result screen with replay or menu options.

The strongest technical lesson from the final testing cycle is that the suite is substantial, but not perfect. The weaker areas are:

\begin{itemize}
    \item \textbf{Mock and scene fidelity.} Several tests are scene-coupled component tests rather than isolated pure unit tests.
    \item \textbf{Assertion depth.} UI-facing tests prove structure and state transitions, but they cannot fully prove visual readability or overlay composition.
    \item \textbf{Coverage concentration.} `BattleUI` remains the lowest-covered runtime class in the archived report.
    \item \textbf{PlayMode breadth.} PlayMode coverage is intentionally narrow and proves readiness more than full end-to-end variety.
\end{itemize}

The project is therefore well validated for a compact vertical slice, but the testing package is stronger on deterministic logic than on fully automated presentation proof.
