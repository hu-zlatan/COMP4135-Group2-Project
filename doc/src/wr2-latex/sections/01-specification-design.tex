\chapter{Specification and Design Decisions}

This report focuses on how the \projectname{} concept moved from the WR1 proposal into a constrained, demonstrable Unity prototype. The emphasis of WR2 is not to restate the full WR1 requirements, but to validate which requirements survived contact with implementation, which ones were simplified, and which ones were deliberately deferred to later stages. The report follows the course specification and the recovered WR2 template structure rather than the incorrect WR1-flavoured Moodle \texttt{main.tex} skeleton \cite{moduleSpecification,wr2Template}. It is intentionally anchored to one frozen prototype snapshot, recorded in the project record \path{wr2-snapshot-freeze-2026-04-02.md}. This means the chapter describes the current demo as it exists in the frozen WR2 evidence set, and separates target-state ambitions from implemented behaviour.

\section{Requirements Validation}

The current prototype validates the core loop of selecting a unit, drawing cards, playing a limited number of command cards, ending the turn, and observing an enemy response. However, several presentation and completion-oriented requirements remain partial rather than finished. The key WR1 to WR2 validation outcomes are:

\begin{itemize}
\item \textbf{6x6 tactical battle board.} Status: retained and implemented. Evidence: the current Unity scene and prototype screenshots. Reason: a single 6x6 scene is large enough to demonstrate positioning without expanding into campaign-scale scope.
\item \textbf{Hand zone, turn controls, and combat information in one interface.} Status: retained and implemented. Evidence: the current \texttt{BattleUI} panel and screenshots. Reason: a left-side control panel plus in-world board keeps the prototype readable for reporting and demonstration.
\item \textbf{Move, Strike, Guard, and Push as the starter card set.} Status: retained and narrowed. Evidence: the fallback deck and explicit card resolution logic. Reason: these four cards cover movement, damage, defense, and displacement with minimal rules overhead.
\item \textbf{Card draw, discard, and fixed plays per turn.} Status: retained and implemented. Evidence: current turn flow and deck management logic. Reason: three cards drawn per turn and two playable actions proved easier to demonstrate than a wider energy or deckbuilding model.
\item \textbf{Enemy units perform visible actions during the enemy phase.} Status: retained but simplified. Evidence: deterministic enemy movement and attack summary output. Reason: the prototype validates enemy response without requiring a broader AI system.
\item \textbf{Clear visual and animation feedback for all actions.} Status: modified and partial. Evidence: tile highlights, card tinting, selection state, and status log. Reason: bespoke combat animation was reduced to lightweight prototype cues to keep WR2 scope under control.
\item \textbf{Win/lose handling ends the combat session cleanly.} Status: retained and implemented in minimal form. Evidence: the current turn manager now stops the battle loop when one side is eliminated, and the battle UI surfaces a textual victory/defeat outcome. Reason: the team fixed the earlier accuracy gap with a narrow runtime change, while leaving richer result presentation for later polish.
\item \textbf{Minimal onboarding cues for first interactions.} Status: partial. Evidence: the current UI gives a status prompt such as ``Select a player unit,'' but not a fuller guided onboarding flow. This gap was confirmed during WR2 validation.
\item \textbf{Structured repository workflow and traceable progress records.} Status: partially implemented and now evidenced through the cleaned GitHub workflow. Evidence: merged pull requests on the canonical repository, the collaboration-history note, and the process evidence package under \path{doc/src/project-records/process/}. Reason: the repository workflow is now visible in GitHub history, but task-board captures and stronger attendance evidence are still weaker than the repository trail itself.
\end{itemize}

\subsection{Updated Functional and Nonfunctional Priorities}

Implementation narrowed the functional priority to one starter deck and one readable combat loop. The retained starter card set is therefore \texttt{Move}, \texttt{Strike}, \texttt{Guard}, and \texttt{Push}. The team explicitly did \emph{not} expand WR2 into a larger card pool or more experimental mechanics such as persistent status effects, directional bonuses, or health-for-power exchanges, because those would require extra balancing and testing effort that the current report cannot honestly evidence.

The most visible remaining nonfunctional issue is readability. The current build exposes health, guarding, prompts, and turn outcome information well enough for demonstration, but the combat HUD is still lighter than a finished interface. Clearer presentation of unit state, stronger explanation of board highlights, and more polished result feedback remain follow-up quality targets rather than completed WR2 claims.

\subsection{Key Design Trade-offs}

The main trade-off in WR2 was breadth versus stability. A broader deck and more specialised combat roles might improve longer-term gameplay depth, but they would also multiply balancing effort and make the prototype harder to validate in a short coursework cycle. The implemented choice was therefore conservative: keep the card set small, keep enemy behaviour deterministic, and make the runtime responsibilities easy to trace. This trade-off supports the real goal of WR2, which is to present a truthful playable slice rather than a larger but weakly evidenced design.

The validated WR2 scope is therefore narrower than the original concept sketched in WR1 and the internal one-pager stored at \path{doc/src/onepager_en.md}. The frozen prototype snapshot commits to a single battle scene, two player units, one enemy response loop, four starter cards, clear board-state feedback, and truthful battle-end termination with text-based outcome reporting. Features such as a larger card pool, multi-scene structure, deckbuilding, progression, and a more expressive tutorial flow remain outside the current implementation window described in the internal architecture note \path{doc/src/demo_architecture_constraints.md}. They are discussed as deferred work rather than being blended into the current-state description.

\section{Architectural/System Diagram}

The current system architecture follows the ``single battle, shallow object model'' direction set out in the internal architecture constraints and first-pass script notes stored at \path{doc/src/demo_architecture_constraints.md} and \path{doc/src/demo_first_pass_scripts.md}. \caption{Current WR2 runtime architecture for the \projectname{} prototype}
\label{fig:architecture}
\end{figure}

The control flow begins at \texttt{GameRoot}, which wires together the scene managers and registers the units already placed in the scene. \texttt{TurnManager} owns turn state, remaining card plays, and the switch between player and enemy phases. \texttt{BattleUI} is intentionally thin: it handles card selection, world clicks, and status output, but it delegates gameplay decisions to the deck, grid, and card systems.

The data flow is intentionally conservative. \texttt{DeckManager} owns draw, hand, and discard state. \texttt{CardResolver} is the only module allowed to interpret \texttt{CardType}, which keeps card behaviour explicit and easy to trace. \texttt{GridManager} owns tile generation, occupancy, and reachability, while \texttt{UnitController} stores runtime unit state such as hit points, team, guarding, and grid position. This arrangement matches the current prototype goal: one readable vertical slice rather than a general-purpose tactics engine.
