\section{Requirements Validation}

WR1 defined a small tactical card battle demo: a card-driven turn loop, a bounded board, visible enemy actions, readable combat state, and enough documentation to justify implementation choices. WR2 validates that baseline against the current prototype. Some requirements now work as intended, while others were narrowed so the demo stayed playable within the coursework scope.

The current prototype validates the core loop of selecting a unit, drawing cards, playing a limited number of command cards, ending the turn, and observing an enemy response. Some presentation and completion-related requirements are still partial. The main WR1 to WR2 validation outcomes are:

\begin{itemize}
\item \textbf{6x6 tactical battle board.} Status: retained and implemented. Evidence: the current Unity scene and prototype screenshots. Reason: a single 6x6 scene is large enough to demonstrate positioning without expanding into campaign-scale scope.
\item \textbf{Hand zone, turn controls, and combat information in one interface.} Status: retained and implemented. Evidence: the current \texttt{BattleUI} panel and screenshots. Reason: a left-side control panel plus in-world board keeps the prototype readable for reporting and demonstration.
\item \textbf{Move, Strike, Guard, and Push as the starter card set.} Status: retained and narrowed. Evidence: the fallback deck and explicit card resolution logic. Reason: these four cards cover movement, damage, defense, and displacement with minimal rules overhead.
\item \textbf{Card draw, discard, and fixed plays per turn.} Status: retained and implemented. Evidence: current turn flow and deck management logic. Reason: three cards drawn per turn and two playable actions proved easier to demonstrate than a wider energy or deckbuilding model.
\item \textbf{Enemy units perform visible actions during the enemy phase.} Status: retained but simplified. Evidence: deterministic enemy movement and attack summary output. Reason: the prototype validates enemy response without requiring a broader AI system.
\item \textbf{Clear visual and animation feedback for all actions.} Status: modified and partial. Evidence: tile highlights, card tinting, selection state, and status log. Reason: bespoke combat animation was reduced to lightweight prototype cues to keep WR2 scope under control.
\item \textbf{Win/lose handling ends the combat session cleanly.} Status: retained and implemented in minimal form. Evidence: the current turn manager now stops the battle loop when one side is eliminated, and the battle UI surfaces a textual victory/defeat outcome. Reason: the team fixed the earlier accuracy gap with a narrow runtime change, while leaving richer result presentation for later polish.
\item \textbf{Minimal onboarding cues for first interactions.} Status: partial. Evidence: the current UI gives a status prompt such as ``Select a player unit,'' but not a fuller guided onboarding flow. This gap was confirmed during WR2 validation.
\item \textbf{Structured repository workflow and traceable progress records.} Status: partially implemented and now evidenced through the cleaned GitHub workflow. Evidence: merged pull requests on the canonical repository, the collaboration-history note, and the process evidence pack listed in Appendix~\ref{sec:supporting-index}. Reason: the repository workflow is now visible in GitHub history, but task-board captures and stronger attendance evidence are still weaker than the repository trail itself.
\end{itemize}

\subsection{Updated Functional and Nonfunctional Priorities}

Implementation kept the functional scope to one starter deck and one readable combat loop. The retained starter card set is \texttt{Move}, \texttt{Strike}, \texttt{Guard}, and \texttt{Push}. WR2 does not expand into a larger card pool or more experimental mechanics such as persistent status effects, directional bonuses, or health-for-power exchanges, because those would need more balancing and testing than the current report can support.

The main remaining non-functional issue is readability. The current build already shows the most important state in visible places: unit hit points and guard state can be checked during play, the status prompt tells the player when to select a unit or target, and battle-end text appears in both the top bar and the combat log. However, the combat HUD is still basic. Board highlight meanings are not fully explained, and the result presentation still relies mainly on text.

\subsection{Key Design Trade-offs}

The main trade-off in WR2 was breadth versus stability. A larger deck and more specialised combat roles could add depth, but they would also increase balancing work and testing risk. The team therefore kept the card set small, kept enemy behaviour deterministic, and kept script responsibilities easy to trace. This supports the main WR2 goal: a working slice that the report can describe clearly.

The WR2 scope is smaller than the broader WR1 concept and one-pager listed in Appendix~\ref{sec:supporting-index}. The frozen snapshot covers one battle scene, two player units, one enemy response loop, four starter cards, board-state feedback, and battle-end text feedback. Larger card pools, multi-scene flow, deckbuilding, progression, and a fuller tutorial sit outside the current implementation window described in the architecture-constraint note in Appendix~\ref{sec:supporting-index}. They are treated as deferred work.

The current architecture follows the single-battle direction set out in the architecture note and first-pass script list in Appendix~\ref{sec:supporting-index}. Figure~\ref{fig:architecture} shows the runtime modules used by the demo. An architecture diagram is used here because this section needs to show how scene managers, UI, cards, grid, and units interact at runtime.

\label{fig:architecture}

The control flow begins at \texttt{GameRoot}, which wires together the scene managers and registers the units already placed in the scene. \texttt{TurnManager} owns turn state, remaining card plays, and the switch between player and enemy phases. \texttt{BattleUI} stays small: it handles card selection, world clicks, and status output, then passes gameplay decisions to the deck, grid, and card systems.

The data flow is simple. \texttt{DeckManager} owns draw, hand, and discard state. \texttt{CardResolver} is the only module that interprets \texttt{CardType}, which keeps card behaviour explicit. \texttt{GridManager} owns tile generation, occupancy, and reachability, while \texttt{UnitController} stores runtime unit state such as hit points, team, guarding, and grid position. This matches the current prototype goal: one readable vertical slice.
