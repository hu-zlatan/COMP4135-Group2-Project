The current frozen prototype feature state can be summarised as follows:

\begin{itemize}
\item \textbf{Implemented.} 6x6 battle grid, unit selection, starter hand and card draw, Move/Strike/Guard/Push resolution, enemy response in simplified form, board highlight feedback, and battle-end gating with textual victory/defeat reporting.
\item \textbf{Partial.} Onboarding cues and feedback emphasis. The UI prompts the player. It does not yet teach the full interaction sequence or explain all board highlight meanings. The battle-end result is visible, but still lightweight.
\item \textbf{Initial automated verification.} A committed EditMode test suite checks three battle-end scenarios in the turn manager, giving WR2 a minimal automated validation layer.
\item \textbf{Deferred.} Broader automated coverage, stronger external playtesting, and richer audiovisual result presentation remain for the next reporting cycle.
\end{itemize}

Several implementation constraints shaped the prototype:

\begin{itemize}
\item The UI is a hybrid prototype solution. It favours quick wiring and readable reporting over long-term extensibility.
\item Input uses the current Input System plus physics raycasts. This links button-based card choice to tile and unit selection with little extra infrastructure.
\item Enemy behaviour is deterministic. This keeps debugging, validation, and reporting manageable.
\item The prototype focuses first on the battle loop and feedback model. Later work can improve battle-end presentation, onboarding, and automated coverage.
\end{itemize}

\section{Design Validation}

WR2 validation used an internal heuristic walkthrough of the frozen prototype snapshot on 01/04/2026. The local workspace does not yet preserve a separate user-test archive, so this stage uses structured scenario review. That matches the seminar distinction between early validation and the fuller testing structure expected in WR3 \cite{week6Seminar}. The walkthrough checked three scenarios aligned with the current demo goals: selecting a player unit, playing a card against a valid target, and ending the turn to observe enemy behaviour. The note is listed in Appendix~\ref{sec:supporting-index} as \texttt{validation/wr2-heuristic-walkthrough-2026-04-01.md}. After that walkthrough, one narrow runtime fix and an EditMode test pass were added to correct the battle-end accuracy gap without widening the prototype scope.

\label{fig:end-smoketest}

Figure~\ref{fig:end-smoketest} adds an automated check alongside the walkthrough evidence. It does not replace broader WR3 testing. It does show that the corrected battle-end behaviour was executed successfully in the Unity Test Framework.

The key validation findings are:

\begin{enumerate}
\item \textbf{Scenario: first interaction and unit selection.} Finding: the player can infer the next step from the left-side status prompt, but there is no stronger onboarding cue on the board itself. Severity: medium. Proposed action: add a clearer first-turn prompt or highlight the expected first click. Status: planned.
\item \textbf{Scenario: play a card and confirm its target.} Finding: card meanings are readable, but the board does not explain move versus attack highlight colours directly. Severity: medium. Proposed action: add a tiny legend or one-line prompt when a card is selected. Status: planned.
\item \textbf{Scenario: end turn and observe enemy response.} Finding: enemy actions were readable through summary text, but the battle loop initially lacked a clear end-state gate and outcome cue. Severity: high. Proposed action: add explicit battle-end termination and visible victory/defeat feedback before final WR2 closeout. Status: applied in narrow form through the prototype-accuracy fix; stronger post-action emphasis remains planned.
\end{enumerate}

These findings shaped the final wording of WR2. The report claims a working vertical slice with battle-end gating, text outcome feedback, and an initial automated check. It does not claim a finished tutorial flow, a polished result screen, or full audiovisual combat feedback.
