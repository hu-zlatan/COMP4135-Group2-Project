\subsection{Gameplay and Feature Walkthrough}

The prototype and feature set can be explained in a direct reader-facing way as follows:

\begin{enumerate}[itemsep=5pt]
\item \textbf{What the player sees.} At battle start, the player sees a small 6x6 board with friendly and enemy units already placed, a left-side battle panel, a top-bar turn summary, and a hand of command cards. Figures~\ref{fig:prototype-evidence} and \ref{fig:prototype-board} show this full scene composition.
\item \textbf{How one turn works.} The player selects a friendly unit, clicks one card from hand, and then clicks a highlighted tile or valid enemy target. This can be repeated until the turn's fixed card-play allowance is exhausted. The player then ends the turn and the enemy performs a simplified response. Figure~\ref{fig:prototype-evidence} shows the board and UI in the same frame so this sequence is understandable.
\item \textbf{What the four starter cards mean.} \texttt{Move} repositions the selected unit onto a highlighted tile, \texttt{Strike} deals direct damage to a valid enemy in range, \texttt{Guard} reduces the next incoming damage to the selected unit by 1, and \texttt{Push} deals light damage and attempts to knock the target back by one tile. Figure~\ref{fig:prototype-cards} shows the hand zone used to trigger these four actions.
\item \textbf{What feedback the system gives.} The prototype highlights legal tiles, keeps the selected unit visible on the board, updates turn and status text, records actions in the log, and reports victory or defeat through lightweight text feedback. Figures~\ref{fig:prototype-evidence} and \ref{fig:prototype-cards} are the main references for these feedback features.
\item \textbf{Unfinished features.} The current build still lacks a stronger tutorial, a highlight legend, richer battle-end presentation, broader automated coverage, and more expressive enemy tactics. Figure~\ref{fig:prototype-board} makes that absence visible because the board is shown without extra onboarding overlays.
\end{enumerate}

Current feature state:

\begin{itemize}
\item \textbf{Implemented.} 6x6 battle grid, unit selection, starter hand and card draw, \texttt{Move}/\texttt{Strike}/\texttt{Guard}/\texttt{Push}, simplified enemy response, board highlights, and battle-end text feedback.
\item \textbf{Partial.} The UI prompts the player, but stronger onboarding and clear highlight explanations are still missing.
\item \textbf{Initial automated verification.} A small EditMode suite checks three battle-end scenarios in the turn manager.
\item \textbf{Deferred.} Broader automated coverage, stronger playtesting, and richer audiovisual result presentation remain for the next cycle.
\end{itemize}

Main implementation constraints:

\begin{itemize}
\item The UI is a hybrid prototype solution that favours quick wiring and readable reporting over long-term extensibility.
\item Input uses the current Input System plus physics raycasts, linking button-based card choice to tile and unit selection with little extra infrastructure.
\item Enemy behaviour is deterministic so debugging, validation, and reporting stay manageable.
\item The prototype prioritises the battle loop and feedback model first; stronger presentation and broader testing are later work.
\end{itemize}

\section{Design Validation}

WR2 validation used an internal heuristic walkthrough of the frozen prototype snapshot on 01/04/2026. This stage therefore uses structured scenario review, which matches the seminar distinction between early validation and the fuller testing structure expected in WR3 \cite{week6Seminar}. The walkthrough checked three scenarios aligned with the current demo goals: selecting a player unit, playing a card against a valid target, and ending the turn to observe enemy behaviour. Appendix~\ref{sec:supporting-index} lists the walkthrough record and related supporting evidence. After that walkthrough, one narrow runtime fix and an EditMode test pass were added to correct the battle-end accuracy gap while keeping the prototype scope stable.

Figure~\ref{fig:end-smoketest} adds an automated check alongside the walkthrough evidence. It complements the WR2 walkthrough and shows that the corrected battle-end behaviour was executed successfully in the Unity Test Framework.

The key validation findings are:

\begin{enumerate}
\item \textbf{Scenario: first interaction and unit selection.} Finding: the player can infer the next step from the left-side status prompt, but there is no stronger onboarding cue on the board itself. Severity: medium. Proposed action: add a clearer first-turn prompt or highlight the expected first click. Status: planned.
\item \textbf{Scenario: play a card and confirm its target.} Finding: card meanings are readable, and the board still lacks a direct explanation of move versus attack highlight colours. Severity: medium. Proposed action: add a tiny legend or one-line prompt when a card is selected. Status: planned.
\item \textbf{Scenario: end turn and observe enemy response.} Finding: enemy actions were readable through summary text, but the battle loop initially lacked a clear end-state gate and outcome cue. Severity: high. Proposed action: add explicit battle-end termination and visible victory/defeat feedback before final WR2 closeout. Status: applied in narrow form through the prototype-accuracy fix; stronger post-action emphasis remains planned.
\end{enumerate}

These findings shaped the final wording of WR2. The report presents a working vertical slice with battle-end gating, text outcome feedback, and an initial automated check. Tutorial flow, result-screen polish, and full audiovisual combat feedback remain later work.
